# Prime Period Theory — Agent Instructions (DOX)

## Project overview

This repository is the canonical knowledge base for **Prime Period Theory (PPT)**,
a descriptive music theory framework grounded in prime-generated periodic
relationships. It contains three distinct layers that agents must understand
before touching any file.

## Three-layer architecture

### Layer 1: OKF knowledge bundle (`okf/`)
Markdown concept pages conforming to Open Knowledge Format (OKF) v0.1.
These are the authoritative source of truth for all PPT concepts.
- **Always has YAML frontmatter** with `type`, `title`, `description`, `tags`, `timestamp`
- **Required `type` values:** `index`, `concept`, `reference`, `glossary`
- **Cross-links use relative markdown links** between concept pages
- **Timestamp format:** YYYY-MM-DD
- Every new concept page must be linked from `okf/index.md` under the correct section

### Layer 2: DOX navigation (`AGENTS.md` files)
This file and child `AGENTS.md` files in each subdirectory. Agents must:
1. Read root `AGENTS.md` before any action
2. Read the child `AGENTS.md` in the specific directory they will edit
3. Update the relevant `AGENTS.md` after meaningful structural changes

### Layer 3: GitHub Pages components (`docs/`)
Static site and interactive components served via GitHub Pages.
Separate from OKF — components are built from OKF content but are not OKF files.
See `docs/AGENTS.md` for component conventions.

## Mandatory agent workflow

Before ANY edit:
1. Read this file
2. Read the child `AGENTS.md` in the target directory
3. Read `okf/index.md` to understand current concept map state
4. Read `README.md` to ensure you understand the overarching project scope

After ANY meaningful change:
1. Update `okf/index.md` if a new concept page was added or renamed
2. Update the local `AGENTS.md` if directory structure changed
3. Update `timestamp` in the frontmatter of any modified OKF file
4. Update `README.md` to reflect any major additions, new tooling, or structural changes to the project

## Core terminology (do not redefine without author approval)

| Term | Meaning |
|---|---|
| **PPT** | Prime Period Theory |
| **Prime family** | The set of intervals/ratios generated by a single prime (2, 3, 5, 7, 11) |
| **Periodicity** | The unifying property of pitch, rhythm, and timbre in PPT |
| **Uniform Solfège** | The base-12 notation layer of PPT |
| **Diacritic system** | The prime-family specific suffix system (e.g., Sub, HalfSub, Base, HalfSup, Sup, Axis) for microtonal inflection |
| **Axis** | The +50 cent midpoint between two 12TET positions; the threshold of Du approximation space; 11-prime entry point |
| **12TET** | Twelve-tone equal temperament; the anchor grid |
| **31 EDO** | Thirty-one equal divisions of the octave; primary microtonal system |
| **72 EDO** | Seventy-two equal divisions; the reference grid for the diacritic system |
| **MusiCoil** | Separate spatial notation system; related but not part of PPT core |

## Philosophical stance (preserve in all content)

PPT is **descriptive, not prescriptive**. All content must reflect this. Never
write as though PPT is the only valid framework or that musicians must use it.
The correct framing: PPT offers a vocabulary for making structural relationships
more legible — as geometry does for visual art.

## Language conventions

All documentation must use Australian English spelling conventions (e.g., "analyse" instead of "analyze", "colour" instead of "color", "conceptualise" instead of "conceptualize", "metre" instead of "meter"). Ensure this consistency across all `.md` and `.astro` files.

## Author voice

Content is written in the first person where the author's perspective is
relevant, and in neutral expository prose for reference material. Avoid
academic hedging ("it could be argued that") but also avoid overclaiming
("PPT proves that"). Aim for the register of a precise, confident working
notebook.

## What agents should NOT do

- Do not modify `okf/index.md` concept map links without also creating the
  linked file
- Ensure diacritic suffix conventions follow the new exact family naming (Sub, HalfSub, HalfSup, Sup, Axis) instead of the legacy (b/eb/p/ep/x) system.
- Do not add 13-limit or higher prime families — the 11-limit ceiling is
  intentional
- Do not add interactive components to `docs/` without a corresponding OKF
  concept page explaining the underlying theory
- Do not rewrite existing OKF pages wholesale — prefer additive edits and
  new linked pages for new content
